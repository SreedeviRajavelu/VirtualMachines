    üß© 1. Linux Processes

    üîπ What They Are

    A process is an instance of a running program. Every time you run a command or application, Linux creates a process with a PID (Process ID).

    Each process has:

    • A parent process (PPID) ‚Äî the process that spawned it.
    • A user who owns it.
    • A state (running, sleeping, stopped, zombie).
    • CPU/memory usage info.

    üîπ Key Commands

    ps ‚Äî process status

    Shows currently running processes.

    ps aux        # show all processes with detailed info
ps -ef        # alternative format, common in system admin
ps -u sree    # show processes by specific user

    Example output fields:

    USER  PID  %CPU  %MEM  VSZ  RSS  TTY  STAT  START  TIME  COMMAND
    • %CPU / %MEM ‚Äî how much CPU/memory process uses
    • STAT ‚Äî process state (R = running, S = sleeping, Z = zombie)

    ‚úÖ Interview tip: If asked ‚Äúhow to find what‚Äôs consuming high CPU?‚Äù, say: ‚ÄúI‚Äôd run ps aux --sort=-%cpu | head or use top for live view.‚Äù

    top ‚Äî dynamic view of running processes

    Real-time system monitor (press q to quit).

    top

    Look for:

    • %CPU, %MEM usage by each process
    • load average (top right): indicates overall system load
    • Tasks summary (running, sleeping, stopped, zombie)

    Useful keys in top:

    • P ‚Äì sort by CPU
    • M ‚Äì sort by memory
    • k ‚Äì kill process (enter PID)
    • 1 ‚Äì show per-CPU usage

    ‚úÖ Scenario example: ‚ÄúServer feels slow ‚Äî I‚Äôd open top to check which process spikes CPU or memory, then investigate that PID with ps -fp <pid>.‚Äù

    üîπ Process Management Commands

    kill <pid>           # terminate a process
kill -9 <pid>        # force kill (SIGKILL)
pkill nginx          # kill all nginx processes
systemctl restart nginx   # restart a service safely

    ‚öôÔ∏è 2. systemd ‚Äî Service and Process Manager

    üîπ What It Is

    systemd is the init system responsible for starting and managing services after boot.

    Services are called units, and they are defined in files ending with .service under /etc/systemd/system or /lib/systemd/system.

    üîπ Common Commands

    systemctl status nginx        # view service status
systemctl start nginx         # start a service
systemctl stop nginx          # stop a service
systemctl restart nginx       # restart a service
systemctl enable nginx        # enable on boot
systemctl disable nginx       # disable on boot
systemctl list-units --type=service   # list active services

    ‚úÖ Interview tip: ‚ÄúIf a web app doesn‚Äôt respond, I‚Äôd check if its service is active using systemctl status <service>, review its logs using journalctl -u <service>.‚Äù

    üßæ 3. Logs and journalctl

    üîπ Where Logs Live

    Linux logs are in /var/log. Common ones:

    /var/log/syslog        # system messages (Ubuntu/Debian)
/var/log/messages      # general logs (CentOS/RHEL)
/var/log/auth.log      # login attempts, sudo usage
/var/log/nginx/error.log  # web server logs

    üîπ journalctl

    Used with systemd to view logs.

    journalctl -xe                   # view latest logs with explanations
journalctl -u nginx.service      # view logs for specific service
journalctl --since "1 hour ago"  # filter by time
journalctl -f                    # follow logs in real time (like tail -f)

    ‚úÖ Scenario example: ‚ÄúWhen debugging a failed service restart, I‚Äôd run systemctl status and journalctl -u <service> to find detailed failure logs.‚Äù

    üîê 4. Permissions and Ownership

    üîπ Linux Permission Model

    Every file has:

    • Owner
    • Group
    • Others

    Permissions:

    • r ‚Äì read
    • w ‚Äì write
    • x ‚Äì execute

    üîπ Commands

    ls -l                 # view permissions
chmod 755 filename    # set permissions (rwxr-xr-x)
chown user:group file # change ownership
sudo                  # run command as superuser

    üîπ Example

    -rwxr-xr--  1 sree  devteam  1024 Oct 18 10:00 script.sh

    ‚Üí Owner can read/write/execute ‚Üí Group can read/execute ‚Üí Others can only read

    ‚úÖ Scenario example: ‚ÄúA script fails with ‚ÄòPermission denied‚Äô. I‚Äôd check permissions with ls -l and use chmod +x script.sh to make it executable.‚Äù

    üíª 5. Resource Monitoring

    üîπ Commands

    Command Purpose
    top View CPU, memory, and load average in real time
    free -h Check memory usage (RAM, swap)
    df -h Check disk space usage
    du -sh /var/log Check folder size
    uptime Show system uptime and load average
    vmstat CPU, memory, I/O stats
    iostat Disk I/O performance
    sar Historical performance (requires sysstat package)

    ‚úÖ Scenario: ‚ÄúIf a service is lagging, I‚Äôd check CPU/memory usage via top, disk space via df -h, and whether swap usage is high via free -h.‚Äù

    üîç 6. Useful Diagnostic Tools

    lsof ‚Äî List Open Files

    Shows what files or network sockets a process has open.

    lsof -p <pid>           # files opened by specific process
lsof -i :80             # processes using port 80

    ‚úÖ Use Case: ‚ÄúPort 8080 is in use ‚Äî I‚Äôd run lsof -i :8080 to find which process occupies it.‚Äù

    netstat or ss ‚Äî Network Connections

    Show network sockets and listening ports.

    netstat -tulnp          # list all listening ports with process IDs
ss -tuln                # modern alternative to netstat
    Option Meaning
    -t TCP connections
    -u UDP connections
    -l Listening ports
    -n Show numeric addresses
    -p Show PID/program name

    ‚úÖ Scenario: ‚ÄúTo verify if Nginx is listening on port 443, I‚Äôd use ss -tulnp | grep 443.‚Äù

    üìä 7. Load Average

    üîπ What It Means

    Appears at the top of top or via uptime:

    load average: 1.24, 0.87, 0.75

    These are average runnable processes over 1, 5, and 15 minutes.

    Interpretation:

    • If you have 4 CPU cores and a load average of 1.0 ‚Üí system is under 25% load.
    • Load average ‚âà number of processes waiting for CPU.
    • High load (> number of cores) means CPU is bottlenecked or too many I/O waits.

    ‚úÖ Scenario example: ‚ÄúIf load average stays above 8 on a 4-core server, I‚Äôd check CPU utilization in top and disk I/O using iostat or vmstat to see what‚Äôs causing the wait.‚Äù

    üß© 1. System Resource Usage Commands

    top ‚Äî Monitor processes and resource usage in real time

    ‚úÖ Purpose: See which processes are consuming CPU, memory, and how the system load changes dynamically.

    top

    Key output fields:

    • %CPU, %MEM ‚Üí show process resource usage
    • load average: ‚Üí number of runnable processes (system load)
    • Tasks: ‚Üí running/sleeping/zombie processes

    Interactive keys:

    • P ‚Üí sort by CPU
    • M ‚Üí sort by memory
    • k ‚Üí kill a process (enter PID)
    • 1 ‚Üí show per-CPU stats

    Example: If the system feels slow, look for a process consuming >90% CPU or memory. You might say in the interview:

    ‚ÄúI‚Äôd run top to check for processes consuming high CPU or memory and use k to kill runaway ones.‚Äù

    free -m ‚Äî Show memory usage

    ‚úÖ Purpose: Display total, used, and available RAM.

    free -m
    • -m ‚Üí show memory in megabytes (-h for human-readable)

    Output example:

                  total    used    free   shared  buff/cache  available
Mem:           7972    2564    1504      120        3904       5120
Swap:          2048       0    2048

    Interpretation:

    • used ‚Üí memory actively in use
    • buff/cache ‚Üí memory used for caching (reclaimable)
    • available ‚Üí memory apps can still use

    ‚úÖ ‚ÄúIf memory usage is near 100% and swap is active, it‚Äôs likely a memory leak or overloaded system.‚Äù

    iostat ‚Äî Monitor disk I/O (input/output performance)

    ‚úÖ Purpose: Identify I/O bottlenecks ‚Äî slow disks or too many reads/writes.

    Install (if missing):

    sudo apt install sysstat

    Run:

    iostat -x 2 3
    • -x ‚Üí extended stats
    • 2 3 ‚Üí update every 2 seconds, 3 times

    Key columns:

    • %util ‚Üí how busy the disk is (close to 100% = bottleneck)
    • r/s, w/s ‚Üí read/write requests per second
    • await ‚Üí average wait time per I/O request

    ‚úÖ ‚ÄúIf %util > 90%, the disk is saturated ‚Äî I‚Äôd check which process is writing heavily using iotop or lsof.‚Äù

    vmstat ‚Äî System performance overview (CPU, memory, I/O)

    ‚úÖ Purpose: Gives a quick view of CPU usage, swap activity, and system processes.

    vmstat 2 5
    • Every 2 seconds, show stats 5 times.

    Key columns:

    • r ‚Üí runnable processes
    • b ‚Üí processes in uninterruptible sleep (I/O wait)
    • si/so ‚Üí swap in/out (should be 0 ideally)
    • us/sy ‚Üí CPU time spent in user/system mode
    • wa ‚Üí time spent waiting on I/O

    ‚úÖ ‚ÄúIf I see high wa (I/O wait) in vmstat, I know disk I/O is the bottleneck, not CPU.‚Äù

    üßæ 2. Inspect Logs

    Check web server and system logs

    Default paths:

    /var/log/nginx/access.log
/var/log/nginx/error.log
/var/log/syslog

    Common commands:

    tail -n 50 /var/log/nginx/error.log      # show last 50 lines
tail -f /var/log/nginx/access.log        # live log updates
grep "500" /var/log/nginx/access.log     # filter HTTP 500 errors
less /var/log/syslog                     # browse system logs

    ‚úÖ Interview answer example: ‚ÄúIf the app returns 500 errors, I‚Äôd check /var/log/nginx/error.log and use grep to filter out error patterns.‚Äù

    üåê 3. Test Connectivity and DNS

    ping ‚Äî Check if a host is reachable

    ‚úÖ Purpose: Test basic network connectivity using ICMP.

    ping google.com
    • Sends ICMP echo requests; reports response time and packet loss.
    • Ctrl + C to stop.

    Output example:

    64 bytes from 8.8.8.8: icmp_seq=1 ttl=115 time=14.3 ms

    ‚úÖ ‚ÄúIf ping fails, it could be a DNS issue, routing problem, or firewall blocking ICMP.‚Äù

    curl ‚Äî Test HTTP(S) connectivity

    ‚úÖ Purpose: Simulate web requests and see responses (great for web app debugging).

    curl -I https://example.com
    • -I ‚Üí fetch headers only
    • -v ‚Üí verbose mode
    • -L ‚Üí follow redirects

    Example:

    curl -v http://localhost:8080

    ‚úÖ ‚ÄúIf the app returns 500, I‚Äôd curl the endpoint to check if backend is reachable or misconfigured.‚Äù

    traceroute ‚Äî Trace network path to a destination

    ‚úÖ Purpose: Identify where packets are delayed or dropped along a route.

    sudo apt install traceroute
traceroute google.com

    Shows each router hop with latency times.

    Interpretation: If delay starts increasing after a certain hop, that‚Äôs where the problem lies (network congestion or bad route).

    ‚úÖ ‚ÄúIf latency is high, I‚Äôd use traceroute to pinpoint where in the path the delay starts.‚Äù

    dig ‚Äî DNS lookup utility

    ‚úÖ Purpose: Query DNS servers to resolve domain names.

    dig google.com

    Output fields:

    • ANSWER SECTION ‚Üí IP address (A or AAAA record)
    • Query time: ‚Üí how long it took
    • SERVER: ‚Üí which DNS server replied

    Example:

    dig @8.8.8.8 example.com +short

    ‚úÖ ‚ÄúIf DNS seems broken, I‚Äôd run dig to see if the domain resolves and which DNS server responds.‚Äù

    nslookup ‚Äî Simpler DNS tool

    ‚úÖ Purpose: Same as dig but simpler format.

    nslookup google.com

    Output:

    Server:  8.8.8.8
Address: 8.8.8.8#53
Name:    google.com
Address: 142.250.72.206

    ‚úÖ ‚ÄúI use nslookup or dig to confirm if DNS is returning the right IP before suspecting network issues.‚Äù

    üîí 4. Check Open Ports and Active Connections

    netstat -tulpen ‚Äî Show active network ports and processes

    ‚úÖ Purpose: See which processes are listening on which ports.

    sudo netstat -tulpen
    Option Description
    -t TCP connections
    -u UDP connections
    -l Listening sockets
    -p Show PID/program
    -e Extended info
    -n Don‚Äôt resolve hostnames

    Output example:

    Proto Recv-Q Send-Q Local Address  Foreign Address  State  PID/Program name
tcp   0      0 0.0.0.0:80         0.0.0.0:*        LISTEN 1234/nginx

    ‚úÖ ‚ÄúTo verify if Nginx is running, I‚Äôd use netstat -tulpen | grep 80.‚Äù

    ss ‚Äî Modern replacement for netstat

    ‚úÖ Purpose: Faster, newer way to view sockets.

    sudo ss -tulnp

    Shows all listening ports and which service owns them.

    ‚úÖ ‚ÄúI‚Äôd use ss -tulnp instead of netstat on modern systems ‚Äî it‚Äôs faster and doesn‚Äôt require extra packages.‚Äù

    üê≥ 5. Container Resource Monitoring

    docker stats ‚Äî Monitor container CPU/memory usage

    ‚úÖ Purpose: Real-time resource stats for running containers.

    docker stats

    Output example:

    CONTAINER ID   NAME         CPU %     MEM USAGE / LIMIT
a1b2c3d4e5f6   nginx_web    2.35%     120MiB / 2GiB
b7c8d9e0f1g2   db_service   10.52%    450MiB / 4GiB

    ‚úÖ ‚ÄúIf a containerized app is slow, I‚Äôd check docker stats to see if a container is hitting its resource limits.‚Äù

    Bonus Commands for Containers

    docker ps                       # list running containers
docker logs <container_id>       # view container logs
docker inspect <container_id>    # show container config/network info
docker exec -it <container_id> bash  # open shell inside container

    üß† 6. Putting It All Together ‚Äî Interview Scenario Example

    Scenario: Users report your web app is slow.

    Your investigation flow (and what to say in interview):

    4. Check DNS resolution: dig app.company.com +short

    5. Verify open ports: ss -tulnp | grep 8080 ‚Üí ensure app is listening

    6. If containerized: docker stats ‚Üí check resource use docker logs <id> ‚Üí application errors

    ‚úÖ What the interviewer sees: You follow a logical, layered troubleshooting process ‚Äî from system ‚Üí network ‚Üí app ‚Üí container ‚Äî just like a real SRE.
